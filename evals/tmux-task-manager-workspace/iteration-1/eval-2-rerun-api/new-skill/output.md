按这个 skill 的约定，应该这样做：

1. 先复用项目级 tmux session
   - 优先使用 `PI_TMUX_SESSION`
   - 如果没有，再用 `./tmux-session-name.sh` 计算 session 名

2. 这次把 API 服务固定放在有意义的窗口名里
   - 用 `api-server`
   - 因为这是同一个逻辑任务，后续重跑也继续用这个名字，不要每次起新名字

3. 重新跑时，不要并存两个同类后台任务
   - 如果已经有同名窗口 `api-server`，应当直接 replace 旧窗口
   - 不要再创建一个新的 `api-server-2`、`api-new` 之类的重复窗口
   - 这样可以保持 “一个逻辑任务 = 一个 tmux window”

4. replace 的原则
   - 先确认目标是同一个逻辑任务
   - 然后用同名窗口替换旧任务，而不是保留旧窗口再额外启动一个新窗口
   - 如果是在排障，先 inspect 旧窗口输出，再重启

5. 为了便于观测退出状态
   - 重跑后仍然保留同一个窗口名 `api-server`
   - 任务退出后尽量保留可观察性，不要让它悄悄消失得无从判断
   - 后续应基于这个固定窗口去看：它是启动成功、正常退出、失败退出，还是在等待输入
   - 一旦窗口已经存在，后续排查时优先按精确窗口目标来 inspect

简化理解就是：
- session 用共享的项目 session
- API 服务窗口名固定为 `api-server`
- rerun 时直接 replace 旧的 `api-server`
- 不制造重复窗口
- 保持这个固定窗口作为之后观察退出状态和排障的统一入口
