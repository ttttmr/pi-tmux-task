这个通知的意思是：某个正在 tmux 里运行的 task 主动触发了终端 bell/notification。基于这个 skill，它本身不等价于“任务成功”或“任务失败”，而是一个“需要查看上下文”的信号。

agent 应该这样处理：

1. 先把它当作检查信号，而不是结果判断
   - 不要因为收到了 terminal notification 就直接宣告完成。
   - 也不要直接把它当成报错。

2. 精确定位对应 task
   - 如果通知里同时有 window id 和 window name，优先用 window id 作为精确引用。
   - 把这个 tmux window 当成对应的那一个逻辑任务来检查，不要新开重复窗口。

3. 立即检查该窗口的上下文输出
   - 看最近日志/终端输出。
   - 判断它是在提示关键事件、等待确认、打印结果，还是发出了警告。

4. 重点判断是否其实卡在交互输入
   - 有些 terminal notification 实际上意味着程序在提醒人看屏幕，比如密码输入、确认提示、选择菜单、异常中断后的确认等。
   - 如果发现是 waiting for input，要按更高优先级处理，因为任务虽然还活着，但已经阻塞了。

5. 根据检查结果再决定后续动作
   - 如果只是普通提醒：向用户简要说明发生了什么。
   - 如果已经明确成功结束：报告完成。
   - 如果暴露出失败迹象：继续按 failed 的方式处理，分析原因，必要时在同一个任务名下重启。
   - 如果需要安全且显然的后续交互：可以执行，或者明确告诉用户下一步该怎么做。

简化来说：

- terminal notification = “这个 tmux task 有值得查看的动静了”
- 不是成功/失败结论
- 正确动作是先 inspect context，再决定是汇报、交互、排障，还是重启
