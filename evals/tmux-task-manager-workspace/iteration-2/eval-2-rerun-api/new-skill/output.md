`api-server` 这个名字是合适的：语义清楚，而且符合“一个逻辑任务对应一个固定 task slot”的约定。

如果你是“重新跑一遍”之前的 api 服务，按这个 skill 的约定应该这样做：

1. 继续使用同一个项目级 task session，不要新开一套临时 session。  
2. 把这个服务继续视为同一个逻辑任务，任务名就固定用 `api-server`。  
3. 如果之前已经有同名后台任务，优先复用这个已有的 `api-server` task slot，而不是再创建一个重复的新任务。  
4. 先检查这个旧的 `api-server` 现在是什么状态：
   - 如果进程还在跑，先对它做一次干净的中断，再在同一个 task slot 里重新启动。
   - 如果它已经退出了，也优先直接在同一个 task slot 里 rerun。
   - 只有在这个旧 slot 明显坏掉、卡死、不可复用时，才替换它；不要并存两个同名或同义任务。

这样做的好处是：

- `api-server` 这个任务身份会保持稳定，后续更容易持续观察。  
- 底层 window id 在可复用时也会尽量保持不变，通知里如果同时出现 window id 和 task name，会更容易对应到同一个服务。  
- 之后无论是正常退出还是异常退出，都更方便根据同一个 `api-server` 任务去看退出状态。

尤其是你提到“便于观测退出状态”，这里的关键约定就是：

- 不要每次 rerun 都新建一个名字不同的任务。  
- 尽量一直复用 `api-server` 这个固定 task slot。  
- 这样后续看到“退出成功 / 失败 / 等待输入 / 消失”等状态时，能明确知道就是这一个 api 服务。  
- 如果任务失败，再去检查输出，并在同一个 `api-server` 名下重启，最利于追踪问题和对照退出码。

总结一下，就是：把任务名定为 `api-server`，如果旧任务存在就先检查，再优先在原 slot 内 clean interrupt + rerun；只有旧 slot 不可用时才替换。这样最符合约定，也最方便后续观测退出状态。
