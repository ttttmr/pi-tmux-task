“disappeared from session” 的意思是：之前对应的 tmux 后台任务窗口，已经不在当前项目共享 session 里了。也就是这个任务对应的 window 消失了，系统现在无法再把它当作一个仍然存在的后台任务来跟踪。

基于这个 skill，它通常不直接等价于“任务失败”，而是先要判断这个 window 是不是被有意移除了。

可能发生的情况主要有这几类：

1. 预期行为
- 这个 window 被手动 kill 了
- 为了修复卡死或损坏状态，旧 window 被替换了
- 同一个逻辑任务在重启时没有复用原 window，而是被新的窗口替代了
- 任务本来就不需要继续存在，完成后相关窗口被清理了

2. 非预期行为
- 误杀了 window
- session 被重建或切换，导致原 window 不见了
- 启动/重启流程不规范，创建了重复任务后又删掉了旧的
- 某个本该持续运行的后台任务意外退出后，窗口也一起消失

下一步判断是不是预期行为，按这个 skill 的思路应这样做：

1. 先确认这个 window 是否是被有意 kill 或 replace 的
- 重点先判断：这是正常替换，还是意外消失
- 因为 skill 对 disappeared 的第一反应就是“verify whether the window was intentionally killed or replaced”

2. 看当前 session 里剩余的任务状态
- 是否存在一个新的、语义相同的 window
- 是否只是同一个逻辑任务改名或被替代了
- 是否项目共享 session 还在，只是这个任务窗口没了

3. 结合任务语义判断它“本来是否应该继续存在”
- 如果这是 dev server、watch、监控、长扫描这类应持续运行的任务，那么消失通常需要进一步处理
- 如果这是一个已经完成、且不需要驻留的任务，那么消失可能是正常的

4. 如果不是预期行为，就按原任务名恢复
- skill 的要求是：如果不符合预期，就在同一个 task name 下重启
- 并优先保持“一个逻辑任务 = 一个 window”的稳定映射

可以用一句话概括判断标准：

- 如果这个 window 是被有意 kill/replace，或者这个逻辑任务已经不需要继续存在，那么这可能是预期行为。
- 如果它本应继续在后台运行，但现在 window 无故不见了，那就应视为异常，先检查剩余状态，再按原任务名重启。
